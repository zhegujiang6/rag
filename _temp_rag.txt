智能文档检索助手 — RAG 优化差距分析及评测建议
生成日期：2026年06月30日
对照《RAG优化》文档 vs. 当前项目实际实现
一、概述
本报告对照《RAG优化.docx》中提出的 11 个优化方向，逐一分析当前项目代码的实现状态。项目当前采用 Streamlit + ChromaDB + MySQL 架构，已实现基础 RAG 流程：文档上传 → 解析 → 固定大小切分（父子块策略） → 向量嵌入 → ChromaDB 检索 → LLM 生成回答。以下是各优化方向的详细差距分析。

【RAG优化文档要求】Agent 规划要不要检索。当前完全未实现。系统在 RAG 模式下总是执行检索，没有判断问题是否需要检索的决策层。chat.py 已区分 RAG/普通对话两种模式，但这是用户手动切换，而非 Agent 自动判断。

四、评测系统设计建议
4.1 评测体系总体架构
建议构建三层评测体系：离线评测（Offline）→ 在线评测（Online）→ 人工评测（Human-in-the-loop）。以离线评测为核心，覆盖 RAG 管线的每个环节。
4.2 评测数据集构建
第一步：构建标准评测数据集（建议放在项目 data/eval/ 目录下）
问答对数据集（QA Pairs）：人工标注 50-200 组 {question, answer, relevant_docs} 三元组
文档集：从已上传的文档中选取代表性样本，覆盖不同格式和领域
Query 分类标注：标注 query 类型（事实查询 / 推理 / 摘要 / 对比 / 多跳推理），便于分类评测
难度分级标注：简单（单文档直接匹配）/ 中等（跨文档综合）/ 困难（需要推理）

C. 端到端评测（End-to-End）
将检索 + 生成作为黑盒，评测最终答案质量：
答案准确率（Accuracy）：生成答案与参考答案的匹配度
首字延迟（TTFT - Time To First Token）：从提问到第一个 token 的时间
完整响应时间：从提问到完整回答的总时间
幻觉率（Hallucination Rate）：答案中包含文档未提及信息的比例
拒答率：正确识别"文档中无相关信息"的比例（避免强行编造回答）
4.4 推荐技术方案
方案一：RAGAS 评测框架（推荐优先采用）
RAGAS（RAG Assessment）是专门为 RAG 系统设计的评测框架，无需大量标注数据即可评测。
安装：pip install ragas datasets
集成方式：新建 services/evaluation_service.py，封装 RAGAS 评测流程
需要的输入：question, answer, contexts（检索到的文档片段）
自动计算：Faithfulness, Answer Relevancy, Context Recall, Context Precision, Context Relevancy 等指标
可同时支持 LLM-as-Judge 模式（用更强的模型做评判者）
建议配合少量人工标注的 ground_truth 做对照验证
方案二：自建评测模块
在项目中新建 services/evaluation_service.py 和 pages/evaluation.py（评测管理页面），包含：
评测数据集管理：上传 / 编辑 / 导出 QA 评测集
批量回归测试：对所有 QA 对跑一遍完整 RAG 流程，计算各项指标
分环节指标采集：在 rag_engine 中增加 hook 点，记录每一步的中间结果
评测报告生成：自动生成包含指标对比、趋势图的评测报告
A/B 测试支持：同时跑两组不同配置（如不同 chunk_size、不同 top_k），对比效果
定时评测：通过定时任务或手动触发，定期对知识库跑评测，监控质量变化
4.5 数据库表扩展建议
为支持评测系统，建议新增以下数据库表：
eval_datasets — 评测数据集元信息：id, name, description, created_at
eval_qa_pairs — 问答对：id, dataset_id, question, ground_truth_answer, relevant_doc_ids(JSON), difficulty, category
eval_runs — 评测运行记录：id, dataset_id, config_snapshot(JSON), started_at, completed_at, status
eval_results — 单条评测结果：id, run_id, qa_pair_id, retrieved_doc_ids(JSON), generated_answer, faithfulness_score, answer_relevance_score, context_recall_score, context_precision_score, latency_ms, hallucination_flag
user_feedback — 用户反馈：id, message_id, rating(1-5), feedback_type, comment, created_at

五、推荐实施路线
综合考虑依赖关系和业务价值，建议按以下顺序推进：
第一阶段（1-2 周）：搭建评测系统：RAGAS 集成 + 评测数据集构建。先有尺子，再量优化效果。
第二阶段（2-3 周）：Query 改写 + BM25 混合检索：对召回率提升最直接，且相互独立可并行推进。
第三阶段（2-3 周）：知识工程优化：LLM 语义切分 + 自动打 Tag + Rerank 加权。
第四阶段（2-3 周）：反馈回流 + Agentic RAG：用户反馈收集 + Agent 决策是否检索。
第五阶段（持续）：持续评测、微调、知识库维护，跑出真正的知识飞轮。
核心原则：先评测，后优化。用数据驱动改进方向。评测系统是判断所有优化是否有效的基石，应当最优先建设。
